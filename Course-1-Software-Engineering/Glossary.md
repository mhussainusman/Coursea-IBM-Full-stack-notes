# Software Engineering Glossary

This alphabetized glossary contains many of the terms used in Course 1 along side important industry-recognized terms. These terms are important to recognize when working in the industry and participating in user groups.

| Term | Definition |
|---|---|
| 2-tier | A computing model in which the server hosts, delivers, and manages most of the resources and services delivered to the client. |
| 3-tier | Organizes applications into three logical and physical computing tiers: the presentation tier (user interface), the middle tier (application/business logic), and the data tier (storage and management). |
| Agile | An iterative method of software development. Teams work in cycles, or sprints, and unit testing happens in each sprint. At the end of each sprint, a chunk of working code is released for stakeholder feedback. |
| Alpha release | The first functioning version of the system released to a select group of stakeholders. |
| API Gateway | Routes the API from the client to a service. Orchestration handles communication between services. |
| Application deployment environment | The combination of hardware and software resources required to run an application. |
| Application Programming Interface (API) | Code that allows two programs to communicate. |
| Array | A fixed number of elements of the same type, stored in sequential order, starting from index zero. |
| Assembly language | A low-level programming language that uses a set of symbols to represent machine code. |
| Attributes | An object's properties and data. |
| Backend developers | Deal with everything that happens on the server before the code and data are sent to the client. |
| Behavioral models | Describe what a system does, without explaining how it does it. |
| Beta release | Also called a limited release, given to stakeholders outside of the developing organization. |
| Black-box testing | A method of testing where the tester does not look at source code or internal structure. |
| Boolean expression | A type of programming statement with only two values: "true" or "false." |
| Branching | Logic where a program makes a decision and follows a different set of instructions depending on whether certain conditions are met. |
| Build automation servers | Execute build automation utilities on a scheduled or triggered basis. |
| Build automation utilities | Generate build artifacts, like executables, by compiling and linking source code. |
| Business logic | Dictates things like transaction results and what data is written to and retrieved from a database. |
| Cascading Stylesheets (CSS) | Used to create stylish websites — defines, applies, and manages style characteristics like colors, fonts, designs, and layouts. |
| Chief Technology Officer (CTO) | Oversees all research and development in a company; monitors systems and infrastructure against company needs and budget. |
| Class | The generic version of an object — contains the object's generic attributes (properties and methods). |
| Class diagram | Shows how the classes in an object-oriented design relate to one another. |
| Client-Server | Another term for 2-tier. |
| Compiled language | Uses a compiler to convert source code from the programming language into machine code, compiled into one executable file. |
| Component | An individual unit of encapsulated functionality, part of an application alongside other components. |
| Component-based architecture | Focuses on decomposing a design into logical, loosely coupled, independent components that work together. |
| Conditional statement | An "if-then" statement that allows computers to use branching logic. |
| Constant | A data item whose value does not change within a program. |
| CI/CD | Continuous Integration / Continuous Delivery (or Deployment). CI ensures code components work together; CD automatically deploys changes to a testing or staging environment. |
| CRUD | Shorthand for the most common database query types: Create, Read, Update, Delete. |
| Database Management System (DBMS) | Controls a database by connecting it to users or other programs. |
| Distributed system | Multiple services on different machines that coordinate by passing messages via a communication protocol such as HTTP. |
| Driver/navigator | A pair programming style — one developer (driver) types the code while the other (navigator) reviews and gives direction. |
| Dynamic content | Content generated each time it is requested by the client. |
| Encapsulation | Bundling a component's data and methods together to hide its internal state and implementation. |
| Event-driven | A pattern focused on producers (which trigger events) and consumers (which process events), connected via an event router. |
| Extensibility | The ability to add behavior to a component without changing other components. |
| Firewall | Prevents unauthorized access to or from a private network. |
| Framework | A standard way to build and deploy applications — a skeleton you extend with your own code. |
| Frontend developers | Deal with everything that happens on the client side — what the user can see and interact with. |
| Fullstack developers | Have skills, knowledge, and experience in both frontend and backend environments. |
| Function | A piece of structured, reusable code that performs a specific action — takes input, processes it, and returns an output. |
| Functional testing | A type of black box testing concerned with testing inputs and outputs. |
| General availability | A stable version of the software intended for all users. |
| Git | A code repository where you track changes, split code into branches for focused development, and merge them back into the main body of code. |
| Hybrid Cloud | A mix of both public and private clouds working together seamlessly. |
| Hypertext Markup Language (HTML) | Creates the physical structure of a website — text, links, images/videos, dividers, and buttons. |
| Information developer | Another term for technical writer. |
| Inheritance | A subclass inherits the same properties and methods as its parent class, and may add its own additional ones. |
| Integrated Development Environment (IDE) | Makes it easier to build and manage code; supports multiple languages and integrates with tools like Git and GitHub. |
| Integration testing | A testing stage that occurs after unit testing, when components are integrated into a larger product. |
| Interpreted language | Code run through an interpreter on the OS or in a browser, which translates human-readable code into machine code at runtime. Also called a scripting language. |
| JavaScript | An object-oriented language used with HTML and CSS to add interactivity to a website. |
| Leaner Stylesheets (LESS) | Enhances CSS, adding more styles and functions. |
| Library | A collection of reusable code — standard programs and subroutines you can use within your code. |
| Load | The number of concurrent users, transactions, and the amount of data transferred between clients and servers. |
| Load balancer | Distributes network traffic efficiently among multiple servers to prevent overload and maximize availability and responsiveness. |
| Loop | A sequence of instructions that continually repeats until a specific condition is reached. |
| Method | Another term for function. |
| Microservices | An approach to building applications by breaking functionality into modular components called services. |
| Minimal Viable Product (MVP) | A basic feature-set release given to stakeholders to validate assumptions and gather feedback. |
| Node | Any device on a network that can recognize, process, and transmit data to other nodes. |
| Non-functional testing | Tests an application for attributes like performance, security, scalability, and availability. |
| Object-oriented Analysis and Design (OOAD) | An approach for analyzing and designing a software system that will be built using object-oriented programming languages. |
| Object-Oriented Programming (OOP) | A methodology focused on objects (which bundle data and methods together) rather than functions operating on separate data structures. |
| Object-Relational Mapping (ORM) | Tools used to connect a database and retrieve the correct data. |
| On-premises deployment | The organization is responsible for the system, hardware, infrastructure, and maintenance required to run the application. |
| Ops engineer | Another term for site reliability engineer. |
| Package manager | Handles finding, installing, maintaining, or uninstalling software packages on request. |
| Packages | Archive files containing app files, installation instructions, and metadata (description, version, dependencies). |
| Pair programming | Two developers working side-by-side at one computer. |
| Parameter | The data that is passed into a function. |
| Peer-to-peer (P2P) | A decentralized network of nodes that act as both clients and servers, sharing the workload. |
| Ping-pong | A pair programming style using TDD — one developer writes a failing test, the other writes code to pass it, then both refactor together. |
| Principal Engineer | Responsible for the architecture, design, and development of a solution; sets processes and technical direction for the team. |
| Private Cloud | Cloud infrastructure provisioned for exclusive use by a single organization, on-premises or managed by a provider. |
| Procedure | Another term for function. |
| Product manager | Similar to product owner; the term typically used in waterfall or waterfall-like projects. |
| Product owner | Holds the vision for the product, understands client/end-user needs, and leads development to deliver stakeholder value. |
| Production environment | The infrastructure (servers, load balancers, databases) that runs and delivers the application to all end-users. |
| Project manager | Ensures a project runs smoothly — handles planning, scheduling, budgeting, resource allocation, and communication. |
| Prototype | A small-scale replica of the end product used to gather stakeholder feedback, establish requirements, or test basic design ideas. |
| Proxy server | An intermediate server between two tiers, handling requests — used for load balancing, caching, firewalling, encryption, and more. |
| Pseudocode | A beneficial bridge to project code — plain-language steps that closely follow the logic the code will use. |
| Public Cloud | Leveraging supporting infrastructure over the open internet on hardware owned by the cloud provider. |
| QA engineer | Ensures product quality and that requirements are met — writes/executes test cases and reports bugs to the development team. |
| Query | Predefined, understandable instructions used to make a request to a database. |
| Regression testing | Confirms that a recent change (e.g. a bug fix) does not adversely affect existing functionality. Also called maintenance testing. |
| Responsive design | A website automatically resizing to fit the device it's being accessed from. |
| Scalability | An application's ability to dynamically handle growing or shrinking load without affecting performance. |
| Scripting language | Another term for interpreted language. |
| Scrum master | Focused on ensuring team and individual success. |
| Service | A unit of functionality (like a component) designed to be deployed independently and reused by multiple systems, solving a business need. |
| Service-oriented architecture (SOA) | Services are loosely coupled and interface with each other via a communication protocol over a network. |
| Site reliability engineer (SRE) | Bridges development and operations — tracks incidents, automates systems, troubleshoots, and ensures reliability. Also called ops engineer. |
| Soft skills | Personal and interpersonal characteristics — harder to define, quantify, or certify than hard (technical) skills. |
| Software architecture | The blueprint for a software system — defines fundamental structures, component interactions, operating environment, and design principles. |
| Software design document (SDD) | A collection of technical specifications indicating how a design should be implemented — assumptions, dependencies, constraints, requirements, objectives. |
| Software Development Lifecycle (SDLC) | A systematic process to develop high-quality software in a predictable timeframe and budget, meeting client business requirements. |
| Software Requirement Specification (SRS) | A document that combines and lists stakeholders' requirements for an application. |
| Software stack | A combination of technologies (frontend and backend) stacked in a hierarchy that work together to support an application. |
| Squad | A small team of up to 10 developers — typically a squad leader, software engineers, and UX designers. |
| Staging environment | An environment as close to replicating production as possible, but not meant for general users. |
| Stakeholder | Mainly responsible for defining project requirements and providing feedback when clarification is needed. |
| Standard Operating Procedure (SOP) | Written documentation explaining step-by-step how to accomplish a common, organization-specific task. |
| Static content | Content stored on a server, not regenerated per request. |
| Strong style | A pair programming style where the more experienced developer is the navigator, and the junior developer (driver) learns by doing. |
| Structured design | Conceptualizes a software problem into well-organized smaller solution elements called modules and sub-modules. |
| Structured Query Language (SQL) | The most popular language used to query databases. |
| Syntactically Awesome Stylesheets (SASS) | Enables variables, nested rules, and inline imports — faster and easier to write than plain CSS. |
| System architect | Designs and describes a project's architecture and communicates it to the team. |
| System Requirement Specification (SysRS) | Contains system capabilities, interfaces, user characteristics, plus policy, regulation, performance, and security requirements. |
| System testing | A testing stage that occurs once the application is deemed complete. |
| System Under Test (SUT) | The software application while it is being tested. |
| Technical Architect | Another term for principal engineer. |
| Technical lead | Manages a team of developers, responsible for the entire development lifecycle, reports to project stakeholders. |
| Technical writer | Writes documentation for the end-user. |
| Test case | Verifies the functionality of a software application — includes steps, inputs, data, and expected outputs. |
| Unified Modeling Language (UML) diagrams | Diagrams that communicate structure and behavior using common, programming-language-agnostic notation. |
| Unit testing | Tests the smallest isolated component of code, usually done by the developer. |
| Use cases | A description of how a user may interact with the system — more granular than a user story, describing system behavior. |
| User Acceptance Testing (UAT) | A testing stage where the software is tested by the intended user. |
| User experience (UX) designer | Balances intuitive design with robustness — defines how software behaves from the user's perspective. |
| User Requirement Specification (URS) | Describes the business needs and expectations of the end-users of the software system. |
| User stories | A software requirement explained from the end-user's perspective, stated in terms of their goal. |
| Variable | A value that can change, assigned and passed into a function or program depending on conditions. |
| Vector | Similar to an array, but with dynamic size — automatically resizes as elements are added or removed. |
| V-shape model | A sequential method like waterfall, but with paired "verification" and "validation" phases. |
| Waterfall | A sequential method of software development where the output of one phase is the input for the next. |
